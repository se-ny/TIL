# 📝 [TIL] 보안의 핵심: 인증(Authentication)과 인가(Authorization)

보안의 가장 기본적이면서도 중요한 개념인 **인증(AuthN)**과 **인가(AuthZ)**의 정의와 차이점, 그리고 주로 사용되는 방식들을 정리합니다.

---

## 1. 인증 vs 인가 (Authentication vs Authorization)

두 개념은 언뜻 비슷해 보이지만, **'누구인가'**와 **'무엇을 할 수 있는가'**의 명확한 차이가 있습니다.

| 구분 | 인증 (Authentication, AuthN) | 인가 (Authorization, AuthZ) |
| :--- | :--- | :--- |
| **개념** | 시스템이 사용자의 **신원을 확인**하는 과정 | 인증된 사용자가 특정 **권한을 가졌는지 확인**하는 과정 |
| **쉽게 말해?** | "당신은 누구십니까?" (로그인) | "당신은 이 방에 들어올 수 있습니까?" (접근 권한) |
| **예시** | ID/PW 입력, 공인인증서 확인, 생체 인식 | 관리자 페이지 접근, 글쓰기/삭제 권한 확인 |

> 💡 **쉽게 이해하는 비유**
> * **인증:** 회사 로비에서 사원증을 찍고 건물 안으로 들어오는 것. (신원 확인)
> * **인가:** 마케팅팀 직원이 연구소 보안 구역에 들어가려고 할 때, 출입문이 열리지 않는 것. (권한 확인)

---

## 2. 인증(Authentication)의 주요 방식

웹/앱 서비스에서 사용자를 인증하기 위해 주로 사용하는 3가지 방식입니다.

### ① 세션/쿠키 (Session / Cookie) 방식
* **원리:** 사용자가 로그인하면 서버는 메모리(또는 DB)에 세션을 저장하고, 클라이언트에게는 `Session ID`를 쿠키에 담어 보냅니다. 브라우저는 이후 요청마다 이 쿠키를 함께 보냅니다.
* **장점:** 서버가 세션을 제어할 수 있어 강제 로그아웃 등이 가능합니다.
* **단점:** 사용자가 많아지면 서버 메모리 부하가 커지며, 서버를 확장(Scale-out)할 때 세션 공유 설정이 까다롭습니다.

### ② 토큰 기반 (JWT - JSON Web Token) 방식
* **원리:** 인증 성공 시 서버가 사용자의 정보를 담은 암호화된 토큰(JWT)을 발행합니다. 클라이언트는 이 토큰을 저장(Local Storage 등)해두고 요청마다 HTTP 헤더에 담아 보냅니다.
* **장점:** Stateless(서버가 상태를 저장하지 않음)하므로 서버 부하가 적고 확장성이 뛰어납니다.
* **단점:** 토큰이 탈취당하면 만료될 때까지 막기 어렵습니다. (이를 보완하기 위해 Access / Refresh 토큰을 나누어 사용합니다.)
